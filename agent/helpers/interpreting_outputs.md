## Quick Reference

After a MAgPIE run, all results land in `output/:title::date:/` (configured in `config/default.cfg:2298`).
The single most important file is **`fulldata.gdx`** — a GAMS Data Exchange file containing the complete model state dumped at the end of every 5-year timestep (`core/calculations.gms:92`). Everything else — reports, validation PDFs, disaggregated maps — is derived from this file by R post-processing scripts.

**Default output scripts** (`config/default.cfg:2293`):
```r
cfg$output <- c("output_check", "extra/disaggregation", "rds_report")
```

## How Outputs Are Produced

### The Postsolve Pattern
Every module writes outputs in its `postsolve.gms` using an auto-generated `R SECTION` block. For each variable and equation, four attributes are stored (`core/sets.gms:328`):

```gams
*#################### R SECTION START (OUTPUT DEFINITIONS) #####################
 ov_land(t,j,land,"level")    = vm_land.l(j,land);    *** optimal value
 ov_land(t,j,land,"marginal") = vm_land.m(j,land);    *** shadow price
 ov_land(t,j,land,"upper")    = vm_land.up(j,land);   *** upper bound
 ov_land(t,j,land,"lower")    = vm_land.lo(j,land);   *** lower bound
*##################### R SECTION END (OUTPUT DEFINITIONS) ######################
```
*Source: `modules/10_land/landmatrix_dec18/postsolve.gms:25-63`*

**The four attributes:**
- **`level`** = The optimal solution value. This is what you almost always want.
- **`marginal`** = Shadow price / dual value. Shows cost of relaxing a constraint by one unit.
- **`upper`** = Upper bound on the variable.
- **`lower`** = Lower bound on the variable.

### The GDX Dump
After all postsolve phases complete, `Execute_Unload "fulldata.gdx"` (`core/calculations.gms:92`) writes **everything** — all GAMS symbols in memory — into a single GDX. This happens inside the time loop, so fulldata.gdx is overwritten each timestep and only the final-timestep state is fully preserved. However, output parameters like `ov_land(t,j,land,type)` accumulate across all timesteps via the `t` dimension.

### Report Generation
`scripts/output/rds_report.R` calls `magpie4::getReport(gdx)` which reads fulldata.gdx and produces standardized IAMC-format variables in `report.mif` and `report.rds`. Variables follow the format `"Name (unit)"` — the space before the parenthesis is required (`scripts/output/rds_report.R:40-45`).

## Reading GDX Files in R

```r
library(gdx2)
library(magpie4)
gdx <- "output/my_run/fulldata.gdx"

# --- Land use ---
land(gdx, level = "reg")            # Land by type, regional (mio. ha)
land(gdx, level = "cell")           # Land by type, cluster level
land(gdx, level = "glo")            # Global totals
croparea(gdx, level = "reg")        # Crop areas by crop type

# --- Production & Demand ---
production(gdx, level = "reg")      # Agricultural production (mio. tDM)
demand(gdx, level = "reg")          # Total demand by product

# --- Food ---
Kcal(gdx, level = "reg")            # Per-capita calorie intake (kcal/cap/day)

# --- Emissions ---
emissions(gdx, level = "reg")       # GHG emissions
carbonstock(gdx, level = "cell")    # Carbon stocks (mio. tC)

# --- Economics ---
costs(gdx, level = "reg")           # Total costs (mio. USD17MER/yr)

# --- Water ---
water_usage(gdx, level = "reg")     # Water demand

# --- Model diagnostics ---
modelstat(gdx)                      # Model status per timestep

# --- Raw GDX access ---
readGDX(gdx, "ov_land", field = "level")       # Read specific output parameter
readGDX(gdx, "vm_land", field = "level")       # Read variable directly
readGDX(gdx, "p80_modelstat")                  # Read modelstat parameter
```

## Understanding Model Status

`p80_modelstat(t)` records the GAMS solver status for each timestep (`modules/80_optimization/nlp_apr17/solve.gms:95`). The solver retries with different settings if modelstat > 2 (`solve.gms:47-92`).

| modelstat | Meaning | Action |
|-----------|---------|--------|
| **1** | Optimal | ✅ Perfect. LP optimal solution found. |
| **2** | Locally optimal | ✅ Good enough. Expected for NLP solves. **This is the normal result.** |
| **3** | Unbounded | ❌ Problem with objective/constraints. Check cost formulations. |
| **4** | Infeasible | ❌ No feasible solution. See `debugging_infeasibility.md`. |
| **5** | Locally infeasible | ❌ Solver stuck. May need different starting point. |
| **6** | Intermediate infeasible | ⚠️ Solver hit iteration/time limit while infeasible. |
| **7** | Intermediate non-optimal | ⚠️ Solver hit limit but has feasible solution. Results usable with caution. |
| **13** | NA (set by code) | ❌ Solver returned NA; treated as error (`solve.gms:44`). |
| **14** | Initial value | ❌ Solve never ran for this timestep (`solve.gms:10`). |

**Key rule**: modelstat ≤ 2 is acceptable. The script `scripts/output/extra/modelstat.R:50` checks `all(out==2)` and warns on anything else.

If modelstat > 2 and ≠ 7, the model aborts with `"no feasible solution found!"` and dumps `fulldata.gdx` for debugging (`solve.gms:102-107`).

## Comparing Scenarios

**Use `merge_report.R`** (`scripts/output/merge_report.R`) to combine multiple runs:
```bash
Rscript output.R output=merge_report outputdir=output/run1,output/run2
```
This creates `output/report_all.rds` and `output/report_all.mif`.

**Use `comparison_validation.R`** for side-by-side validation PDFs:
```bash
Rscript output.R output=comparison_validation outputdir=output/run1,output/run2
```

**Key variables to compare across scenarios:**
- `ov_land` — Did land allocation change as expected?
- `ov_cost_glo` — What's the cost impact?
- `p80_modelstat` — Are all timesteps feasible?
- Emissions via `oq52_emis_co2_actual` — Carbon impact?
- `ov_prod_reg` — Production shifts?

## Common Pitfalls

1. **Unit confusion**: Costs are `mio. USD17MER/yr` (2017 market exchange rate US dollars). Land is `mio. ha`. Production is `mio. tDM` (dry matter, NOT fresh weight). Carbon stocks are `mio. tC`.

2. **Temporal resolution**: Outputs are at 5-year intervals. Values represent the state AT that timestep, NOT cumulative over the period. Annualized flows (costs, emissions) are per-year rates.

3. **Spatial aggregation**: GDX has cluster-level (`j`, ~200 cells) data. The `report.mif` has regional (`i`, 12 regions) + global. For 0.5° grid, you need the disaggregation output (`cell.land_0.5.mz`) produced by `scripts/output/extra/disaggregation.R`.

4. **Level vs. marginal**: Almost always use `"level"` (the optimal value). Marginals (shadow prices) require careful economic interpretation and are zero for non-binding constraints.

5. **fulldata.gdx overwrites**: The GDX is rewritten each timestep (`core/calculations.gms:92`). The output parameters (`ov_*`, `oq_*`) keep all timesteps via their `t` dimension, but raw variables (`vm_*`) only have the last-timestep values.

6. **`o_` vs `ov_` prefix**: Output declarations use `ov_` for variables and `oq_` for equations. The `o_` prefix is for other output parameters computed in postsolve. The `ov_`/`oq_` blocks are auto-generated (between `R SECTION START/END` markers).

7. **Dry matter vs. fresh weight**: All production in MAgPIE is in dry matter (`tDM`). Conversion factors are in `fm_attributes`. This is a common source of confusion when comparing with FAO statistics (which use fresh weight).

8. **report.mif variable naming**: Variables MUST follow `"Name (unit)"` format with a space before the parenthesis. The `rds_report.R` script warns if this pattern is violated (`scripts/output/rds_report.R:40-45`).
