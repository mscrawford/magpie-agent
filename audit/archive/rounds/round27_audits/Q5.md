### Bug Q5-B1 — `v20_dem_processing` declared over the wrong index set
- **Severity**: Major (tier_uncertainty: false)
- **Class**: 4 (conceptual / scope) — closest §3 class; semantic-scope error analogous to "wrong prefix"
- **Trigger (§1 Major)**: "Wrong variable prefix (semantic scope wrong)" — applied to a wrong index *set*, which narrows the variable's scope.
- **Claim in answer** (line 23, variable table): "`v20_dem_processing` | (i, processing20, kpr) | ... | `declarations.gms:16`".
- **Reality in code**: `declarations.gms:17` — `v20_dem_processing(i,processing_subst20,kpr)`. The index set is **`processing_subst20`** (8 activities: milling, refining, extracting, distilling, fermentation, breeding, ginning, **substitutes**; `sets.gms:20-21`), NOT `processing20` (the 7-member subset that *excludes* substitutes; `sets.gms:23-24`).

So the declaration as stated is wrong vs code AND internally inconsistent with the rest of the answer. A reader trusting the table would conclude the substitutes route cannot be indexed, contradicting the substitution mechanism the answer describes.
- **Source**: answerer_confabulation. The doc (`module_20.md`) does NOT contain this wrong declaration; it uses `v20_dem_processing(...,"substitutes",...)` correctly throughout (module_20.md:148,163,179,203,262). The answerer constructed a declaration row not present in (and contradicted by) its source.
