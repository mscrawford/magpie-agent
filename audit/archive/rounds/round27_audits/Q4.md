### Bugs Found

#### Bug Q4-B1 — Wrong M11 aggregation formula (and stripped "inferred" caveat)
- **Severity**: Major
- **Class**: 4 (Conceptual pseudo-code) / 12 (Content-level citation mismatch)
- **Trigger**: §1 Major — "Fabricated formula… would mislead a reader about how the code reads, but not into a wrong code edit" (matches the R16 `sum(land, vm_land(...))` anchor, scored Major).
- **Claim in answer** (lines 89-92, and summary-table line 114): "Module 11 (Costs) receives it and performs regional aggregation: `sum(j2, vm_cost_transp(j2,k))` summed by region `i`" … "Aggregation pattern (documented in `module_40.md` Interface Variables section): `sum(j2, vm_cost_transp(j2,k))` summed by region `i`".
- **Reality in code**: `modules/11_costs/default/equations.gms:21`:
  ```gams
  + sum((cell(i2,j2),k), vm_cost_transp(j2,k))
  ```
  inside `q11_cost_reg(i2)`. The actual sum runs over BOTH the cell→region mapping `cell(i2,j2)` AND commodities `k`, collapsing both dimensions into a single scalar per region `i2`. The answer's `sum(j2, vm_cost_transp(j2,k))` (i) omits the `cell(i2,j2)` mapping set that does the cell-to-region assignment and (ii) leaves `k` free, implying a per-commodity regional total `X(i,k)` — which is not what the code produces.
- **File evidence**: `modules/11_costs/default/equations.gms:15` (`q11_cost_reg(i2) ..`), `:21` (the `vm_cost_transp` term). Doc source of the wrong formula: `magpie-agent/modules/module_40.md:172`, which is explicitly labelled "**(inferred from standard MAgPIE cost structure)**" at `module_40.md:171`. The answer dropped that caveat and presented it as "documented".
- **Anchor reference**: R16 `q10_land_area` set-sum expansion → Major. Closely analogous (wrong sum structure, real variable names, correct direction).

#### Bug Q4-B4 — Incorrect unit-cancellation rationale
- **Severity**: Minor
- **Class**: 4 (Conceptual pseudo-code, in an explanatory aside)
- **Claim in answer** (line 83): "The factor of 1,000,000 from tDM -> Million tDM cancels with USD -> Million USD, yielding the correct units."
- **Reality**: There is no "USD → Million USD" conversion in the equation.
