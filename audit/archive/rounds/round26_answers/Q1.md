## Default Realization

Module 30's active default realization is `simple_apr24`, set in `config/default.cfg` as `cfg$gms$croparea <- "simple_apr24"` (module_30.md, line 4). The alternative is `detail_apr24`.

---

## Core Area Variable

The primary area variable is `vm_area(j,kcr,w)` (module_30.md, Interface Variables section):

- **Type**: positive variable
- **Dimensions**: `j` (spatial cluster) x `kcr` (19 crop types) x `w` (rainfed/irrigated)
- **Units**: million hectares (mio. ha)
- **Declaration**: `simple_apr24/declarations.gms:21`

This is one of the most widely consumed interface variables in MAgPIE, read by 7 downstream modules. The solver determines its values endogenously subject to constraints.

---

## How `simple_apr24` Allocates Crop Area

The `simple_apr24` realization contains 9 equations. The optimizer allocates `vm_area(j,kcr,w)` by minimizing total system cost subject to:

1. **Production identity** (`q30_prod`, `simple_apr24/equations.gms:14-15`): links area to production.
2. **Rotational diversity constraints** (`q30_rotation_max` and `q30_rotation_min`): hard upper/lower bounds on area shares per crop group, enforced separately for rainfed and irrigated areas.
3. **Bioenergy tree target** (`q30_betr_missing`, `simple_apr24/equations.gms:21-23`): a soft penalty for BETR shortfall.
4. **BETR penalty aggregation** (`q30_cost`, `simple_apr24/equations.gms:25-27`): feeds the penalty into the objective via `vm_rotation_penalty(i)`.
5. **Regional cropland aggregation** (`q30_crop_reg`): aggregates area to regional level for a growth-rate bound.
6. **Carbon and biodiversity tracking** (`q30_carbon`, `q30_bv_ann`, `q30_bv_per`): satellite equations not constraining area directly.

The allocation itself is driven by comparative advantage across cells, mediated by yield differences and costs.

### Key Constraining Equations (simple_apr24)

**q30_prod** (`simple_apr24/equations.gms:14-15`):
```
q30_prod(j2,kcr) ..
  vm_prod(j2,kcr) =e= sum(w, vm_area(j2,kcr,w) * vm_yld(j2,kcr,w));
```
This is the production identity; it does not constrain area directly but ties area to production and thus to the food-supply/demand balance.

**q30_rotation_max** (`simple_apr24/equations.gms:34-36`):
```
q30_rotation_max(j2,crpmax30,w) ..
  sum((crp_kcr30(crpmax30,kcr)), vm_area(j2,kcr,w)) =l=
    sum(kcr, vm_area(j2,kcr,w)) * f30_rotation_max_shr(crpmax30);
```
Hard upper bound: area of each crop group cannot exceed a fixed share of total area within each water supply type (`w`). Shares come from `input/f30_rotation_max.csv`. Set to 1 (inactive) when `c30_rotation_constraints = "off"`.

**q30_rotation_min** (`simple_apr24/equations.gms:42-44`):
```
q30_rotation_min(j2,crpmin30,w) ..
  sum((crp_kcr30(crpmin30,kcr)), vm_area(j2,kcr,w)) =g=
    sum(kcr, vm_area(j2,kcr,w)) * f30_rotation_min_shr(crpmin30);
```
Hard lower bound: ensures minimum diversity. Shares from `input/f30_rotation_min.csv`. Set to 0 when `c30_rotation_constraints = "off"`.

Key distinction from `detail_apr24`: `simple_apr24`'s rotation constraints use a **per-w** formulation (rainfed and irrigated constrained independently, via `crp30` / `crpmax30` / `crpmin30` sets and static CSV shares), whereas `detail_apr24` sums over `w` and uses time-varying scenario faders.

**q30_betr_missing** (`simple_apr24/equations.gms:21-23`):
```
q30_betr_missing(j2)$(sum(ct, i30_betr_penalty(ct)) > 0) ..
  v30_betr_missing(j2) =g=
    vm_land(j2,"crop") * sum(ct, i30_betr_target(ct,j2)) - vm_area(j2,"betr","rainfed");
```
Soft constraint: penalizes shortfall of bioenergy tree (BETR) area relative to a target share of total cropland. Only active after the BETR scenario start year when the penalty rate is non-zero.

**q30_cost** (`simple_apr24/equations.gms:25-27`):
```
q30_cost(i2) ..
  vm_rotation_penalty(i2) =g=
    sum(cell(i2,j2), v30_betr_missing(j2) * sum(ct, i30_betr_penalty(ct)));
```
Aggregates the BETR penalty to region level and feeds it into Module 11's objective function. Default penalty rate: `s30_betr_penalty = 2460 USD17MER/ha`.

**q30_crop_reg** (both realizations):
```
q30_crop_reg(i2) ..
  v30_crop_area(i2) =e= sum((cell(i2,j2), kcr, w), vm_area(j2,kcr,w));
```
Regional aggregation; used in `presolve.gms` to bound cropland growth. Default `s30_annual_max_growth = Inf` (no constraint).

## Connection to Module 10 (Land Pool)

Module 30 does NOT directly write to `vm_land(j,"crop")`. The link runs through **Module 29 (Cropland)**.

Module 29 (default realization `detail_apr24`) defines total cropland via `q29_cropland` (`modules/29_cropland/detail_apr24/equations.gms:11-12`):

```
q29_cropland(j2)..
  vm_land(j2,"crop") =e=
    sum((kcr,w), vm_area(j2,kcr,w)) + vm_fallow(j2) + sum(ac, v29_treecover(j2,ac));
```

(In the `simple_apr24` realization of Module 29 the fallow and treecover terms are absent:
`vm_land(j2,"crop") =e= sum((kcr,w), vm_area(j2,kcr,w));`)

`vm_land(j,"crop")` then enters Module 10's land conservation equation (`q10_land_area`, `modules/10_land/landmatrix_dec18/equations.gms:13-15`):

```
q10_land_area(j2) ..
  sum(land, vm_land(j2,land)) =e= sum(land, pcm_land(j2,land));
```

This enforces that total land (across all 7 types: crop, past, forestry, primforest, secdforest, urban, other) equals the cell's previous-period total. Land cannot be created or destroyed, only converted between types. Cropland expansion thus forces contraction of another land type (typically natural vegetation), which is the mechanism behind deforestation dynamics.

Module 29 also enforces an upper bound on total cropland via `q29_avl_cropland`:

```
q29_avl_cropland(j2)..
  vm_land(j2,"crop") =l= sum(ct, p29_avl_cropland(ct,j2));
```

where `p29_avl_cropland(t,j) = f29_avl_cropland(j,"%c29_marginal_land%") * (1 - p29_snv_shr(t,j))` restricts cropland to biophysically suitable area minus the SNV set-aside share.

**Chain**: `vm_area(j,kcr,w)` [M30] -> `q29_cropland` [M29] -> `vm_land(j,"crop")` -> `q10_land_area` [M10] -> land conservation.

## Connection to Module 14 (Crop Yields)

Module 14 (realization `managementcalib_aug19`) provides `vm_yld(j,kcr,w)` via equation `q14_yield_crop` (`modules/14_yields/managementcalib_aug19/equations.gms:14-16`):

```
q14_yield_crop(j2,kcr,w) ..
  vm_yld(j2,kcr,w) =e= sum(ct, i14_yields_calib(ct,j2,kcr,w))
                        * vm_tau(j2,"crop") / sum((cell(i2,j2), supreg(h2,i2)), fm_tau1995(h2));
```

This scales calibrated LPJmL yields (`i14_yields_calib`) by the ratio of the current technological change factor (`vm_tau(j,"crop")`, from Module 13) to the 1995 baseline (`fm_tau1995`).

Module 30 consumes `vm_yld(j,kcr,w)` in `q30_prod`:

```
vm_prod(j2,kcr) =e= sum(w, vm_area(j2,kcr,w) * vm_yld(j2,kcr,w));
```

The yield is thus exogenous to Module 30 in any given timestep: Module 14 computes it, Module 30 uses it. Because irrigated yields are typically higher than rainfed yields, the optimizer has an economic incentive to allocate irrigated area where irrigation infrastructure exists (constrained by `vm_AEI(j)` from Module 41). The yield differential between crops and cells also drives spatial comparative advantage, determining which crops get grown where.

**Chain**: LPJmL biophysical data + tau (M13) -> `q14_yield_crop` [M14] -> `vm_yld(j,kcr,w)` -> `q30_prod` [M30] -> `vm_prod(j,kcr)` -> Module 17 (aggregation) -> Module 21 (trade) -> food supply/demand balance.
