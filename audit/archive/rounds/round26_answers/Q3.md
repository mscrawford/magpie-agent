## Module 80 - Optimization: Default Realization and Solve Logic

Source: `modules/module_80.md` (verified 2026-05-16 against GAMS source)

---

### 1. Default Realization

The default realization is **`nlp_apr17`**, confirmed in `config/default.cfg` via:
```
cfg$gms$optimization <- "nlp_apr17"
```
(module_80.md, "Related Configuration Files" section)

Three non-default alternatives exist: `lp_nlp_apr17` (LP warmstart + NLP), `nlp_ipopt` (Ipopt interior-point), and `nlp_par` (parallel NLP by superregion).

---

### 2. Objective Function

Module 80 is a **pure solver-orchestration module** — it defines zero model equations. Every equation in the model comes from the other 45 modules.

The model is declared in `main.gms` as:
```gams
model magpie / all - m15_food_demand /;
```
(module_80.md, "Model Declaration" section)

The `nlp_apr17` realization minimizes a single interface variable:

| Variable | Provider | Dimensions | Unit | Description |
|----------|----------|------------|------|-------------|
| `vm_cost_glo` | **Module 11 (Costs)** | (none, global scalar) | 10^6 USD17MER/yr | Total global system costs — the objective function |

The primary solve statement in `nlp_apr17/solve.gms:34` is:
```gams
solve magpie USING nlp MINIMIZING vm_cost_glo;
```

`vm_cost_glo` is provided by Module 11, which aggregates all cost components from across the model into a single global scalar. Module 80 does not define `vm_cost_glo`; it only tells GAMS to minimize it. The dependency chain is: M11 defines `vm_cost_glo` → M80 minimizes it (module_80.md, "Upstream Dependencies"; Module_Dependencies.md).

The `nlp_apr17` realization does NOT perform the land-difference minimization step (`vm_landdiff`) that `lp_nlp_apr17` adds. That secondary minimize is absent from the default realization (module_80.md, "Key Differences from lp_nlp_apr17" table; "Key Limitations" #4).

### 3. Key Parameters and Scalars (nlp_apr17)

Declared in `nlp_apr17/declarations.gms:8-16`:

| Name | Type | Default | Description |
|------|------|---------|-------------|
| `p80_modelstat(t)` | parameter (time) | — | Modelstat indicator per timestep; 1=optimal, 2=locally optimal, >2=infeasible/error |
| `p80_num_nonopt(t)` | parameter (time) | — | Number of non-optimal variables (numNOpt) |
| `s80_counter` | scalar | — | Iteration counter for the retry loop |
| `s80_resolve_option` | scalar | — | Cycles 1-4 across the four fallback strategies |

Declared in `nlp_apr17/input.gms:8-13`:

| Scalar | Default | Description |
|--------|---------|-------------|
| `s80_maxiter` | 30 | Maximum solve iterations before aborting |
| `s80_optfile` | 1 | 0=solver defaults; 1=use `conopt4.opt` |
| `s80_secondsolve` | 0 (off) | If 1, repeats each solve statement immediately |
| `s80_toloptimal` | 1e-08 | CONOPT4 `Tol_Optimality` |

Solver: **CONOPT4**, selected via `option nlp = conopt4;` (nlp_apr17/solve.gms:14). No LP warmstart.

### 4. Infeasibility and Bad Modelstat Handling

The full solve sequence in `nlp_apr17/solve.gms:7-109` is:

**Step 1 — Solver configuration** (solve.gms:13-27):
Sets `option nlp = conopt4`, `option threads = 1`, `magpie.optfile = s80_optfile`, `magpie.scaleopt = 1`, `magpie.solprint = 0`, `magpie.holdfixed = 1`. Initializes `p80_modelstat(t) = 14` (sentinel) and `s80_counter = 0`.

**Step 2 — Primary solve** (solve.gms:33-36):
```gams
solve magpie USING nlp MINIMIZING vm_cost_glo;
if(s80_secondsolve = 1, solve magpie USING nlp MINIMIZING vm_cost_glo; );
```

**Step 3 — Retry loop** (solve.gms:47-93):

If `magpie.modelstat > 2` after the primary solve, the model enters a `repeat...until` loop cycling through four fallback strategies via `s80_resolve_option` (cycles 1-4 then resets to 0):

| `s80_resolve_option` value | Strategy |
|---------------------------|---------|
| 1 | CONOPT4, default settings (`optfile = 0`) |
| 2 | CONOPT4, custom optfile (`optfile = 1`) |
| 3 | CONOPT4, relaxed variable limits (`optfile = 2`, `Lim_Variable = 1.e25`) |
| 4 | CONOPT3 (older, more robust solver; `option nlp = conopt3`) |

Each iteration increments `s80_counter`. The loop pseudocode:
```gams
repeat(
  s80_counter = s80_counter + 1;
  s80_resolve_option = s80_resolve_option + 1;
  // select strategy 1-4
  solve magpie USING nlp MINIMIZING vm_cost_glo;
  s80_resolve_option$(s80_resolve_option >= 4) = 0;
  until (magpie.modelstat <= 2 or s80_counter >= s80_maxiter)
);
```
(nlp_apr17/solve.gms:48-92)

NA modelstat is converted to 13 before the loop condition is evaluated (nlp_apr17/solve.gms:44, 87).

**Step 4 — Finalization** (solve.gms:95-109):

Results recorded:
```gams
p80_modelstat(t) = magpie.modelstat;
p80_num_nonopt(t) = magpie.numNOpt;
```

**On success** (`p80_modelstat(t) <= 2`):
- GDX saved: `mv magpie_p.gdx magpie_YEAR.gdx`
- Model proceeds to the next timestep

**On failure** (`p80_modelstat(t) > 2` AND `p80_modelstat(t) != 7`):
- Debug files zipped: `gmszip -r magpie_problem.zip`
- Full data dumped: `Execute_Unload "fulldata.gdx"`
- Hard abort: `abort "no feasible solution found!"`

Modelstat 7 (feasible but not proven optimal) is accepted as success — excluded from the abort condition. No warning is issued to the user when this occurs (module_80.md, "Key Limitations" #8).

There is no mechanism to skip an infeasible timestep; if all `s80_maxiter` retries fail, the model run terminates unconditionally.

### 5. Modelstat Reference

| Modelstat | Meaning | Module 80 Action |
|-----------|---------|------------------|
| 1 | Optimal | Success — save GDX, continue |
| 2 | Locally optimal | Success — save GDX, continue |
| 7 | Feasible (not proven optimal) | Accepted — treated as success, no warning |
| 13 | Error during solve | Retry (cycles through CONOPT4 configs + CONOPT3) |
| >2 (except 7) | Infeasible/unbounded/error | Retry up to `s80_maxiter` (default 30), then ABORT |
| NA | Not available (evaluation error) | Set to 13, then retry |
