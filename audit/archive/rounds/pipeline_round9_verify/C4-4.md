## Conclusion

Every limb reproduces mechanically:
- The instruction (line 95) tells maintainers to register new scripts into the positive-control list.
- The registration check (line 103) accepts a script on exit-0 alone, discarding output.
- The 6 named scripts exit 0 on `--self-test` with ZERO self-test logic (normal pass over clean
  corpus), so registering any of them asserts "this check is proven-capable" with zero capability proof.
- Positive control: registered scripts have genuine `--self-test` branches + planted fixtures, so the
  exit-0 signal is meaningful only for them — the asymmetry that makes the 6 false controls is real.
- CI blocks on the harness, amplifying the false assurance.

This is a defect, not a removal/merge (no live-dependent question; `live_dependent_found=false`).
Suggested fix correctly carries "Do NOT apply" and is sound: require a `SELFTEST_OK <name>` sentinel
grep (output-based, not exit-code-based) AND have each main() hard-error (exit 2) on `--self-test`
when no self-test is implemented, so a no-op cannot masquerade as a passing control.

**Verdict: UPHELD.**
