#### Bug Q3-B1 — Modelstat 7 mischaracterized as "success" (the audit's flagged critical check)

- **Bug ID**: Q3-B1
- **Severity**: Major
- **Class**: 12 (content-level citation/behavior mismatch) + 4 (conceptual pseudo-code mislabel)
- **Trigger**: §1 Major — "wrong in a way that misleads about behavior, but won't directly cause damaging action." Between Major and Critical → tie-breaker pulls down to Major. `tier_uncertainty: true`.
- **Claim in answer**:
  - §4: "Modelstat 7 (feasible but not proven optimal) is accepted as success — excluded from the abort condition. No warning is issued to the user when this occurs."
  - §5 table row 7: "Accepted — treated as success, no warning" (listed in parallel with rows 1/2 = "Success — save GDX, continue").
- **Reality in code** (`nlp_apr17/solve.gms`):
  1. **Modelstat 7 is `> 2`, so it ENTERS the retry loop** (`if (magpie.modelstat > 2, repeat(...))`, line 47). The loop exit is `until (magpie.modelstat <= 2 or s80_counter >= s80_maxiter)` (line 91). Since 7 is never `<= 2`, a solver that keeps returning 7 burns the **full `s80_maxiter` (30) retries** — it is NOT "accepted" on the primary solve. The answer's "treated as success, continue" omits this entirely.
  2. **No GDX is saved for modelstat 7.** The success/GDX-save check (line 98) is `if ((p80_modelstat(t) <= 2), mv ... magpie_<t>.gdx)`. Modelstat 7 fails `<= 2`, so the timestep GDX is NOT written. The only thing modelstat 7 does at finalization is *avoid the abort* (line 102, `... ne 7`). So 7 is a silent limbo: no abort, no GDX, no warning. Presenting it parallel to rows 1/2 ("save GDX, continue") misleads a reader into thinking a `magpie_<year>.gdx` exists when it does not.
  - The "excluded from the abort condition" sub-claim is the one CORRECT part (line 102 `... and p80_modelstat(t) ne 7`).
- **File evidence**:
  - `modules/80_optimization/nlp_apr17/solve.gms:47` `if (magpie.modelstat > 2,` (7 enters loop)
  - `:91` `until (magpie.modelstat <= 2 or s80_counter >= s80_maxiter)` (7 never satisfies `<= 2`)
  - `:98-99` `if ((p80_modelstat(t) <= 2), ... mv -f magpie_p.gdx ...)` (7 excluded → no GDX)
  - `:102` `if ((p80_modelstat(t) > 2 and p80_modelstat(t) ne 7), ... abort ...)` (7 excluded from abort — the only correct part)
- **Root cause — doc_error**: `magpie-agent/modules/module_80.md:828-840` (Limitation #8) is the source. It claims a "success check" accepting modelstat 7 with the snippet `if ((magpie.modelstat=1 or magpie.modelstat = 7), ...)` cited at "solve.gms:74, 194". **That snippet does not exist in `nlp_apr17/solve.gms`.** It lives in `lp_nlp_apr17/solve.gms:74, 84, 194`, where it is NOT a success/GDX check — it is the **gate for the secondary `vm_landdiff` minimization** (set `vm_cost_glo.up = vm_cost_glo.l`, then `solve ... MINIMIZING vm_landdiff`). The doc mixed a non-default realization's `vm_landdiff` gate into a generic "Module 80 accepts modelstat 7 as success" claim; the answer inherited it.
- **Anchor reference**: resembles the R3 cascading wrong-realization anchor in *origin* (lp_nlp logic described as if it governed the default), but the blast radius here is one behavioral subtlety, not a full cascade → Major, not Critical.
