## The `stockType` dimension

`stockType` is **not declared in Module 52** — it belongs to the interface variables `vm_carbon_stock`/`pcm_carbon_stock`, which are declared in **Module 56** (`modules/56_ghg_policy/price_aug22/declarations.gms:34`). The set itself is declared at `modules/56_ghg_policy/price_aug22/sets.gms:212-213`:

```
stockType Carbon stock types / actual, actualNoAcEst /
```

🟢 It has exactly **two members** — there is no `reference` member (this was a documented correction, R58 2026-07-17, `modules/module_29.md:267-272`). Their meaning is an *age-class-inclusion* axis, not "actual vs. reference":

- **`actual`**: carbon stock summed over **all** age classes, including newly-established ones.
- **`actualNoAcEst`**: carbon stock summed **excluding** the newly-established age-class subset (`ac_sub`), i.e. establishment-year carbon is left out.

The populating equations (e.g. `q29_carbon` for cropland treecover) are indexed over the free `stockType` set and fill **both** slices via a GAMS `$(sameas(...))`-gated branch-selection idiom, not by addition (`modules/module_29.md:249-261`, macro `m_carbon_stock_ac` at `core/macros.gms:104-106`):

```gams
$macro m_carbon_stock_ac(land,carbon_density,sets,sets_sub) \
   sum((&&sets),     land(j2,&&sets)     * sum(ct, carbon_density(ct,j2,&&sets,ag_pools)))$(sameas(stockType,"actual")) + \
   sum((&&sets_sub), land(j2,&&sets_sub) * sum(ct, carbon_density(ct,j2,&&sets_sub,ag_pools)))$(sameas(stockType,"actualNoAcEst"))
```

**Which slice does Module 52 use?** `q52_emis_co2_actual` reads only the **`"actual"`** slice — hardcoded as a literal, not a config switch (`modules/module_52.md:308-310`, confirmed at `cross_module/carbon_balance_conservation.md:101`: *"Module 52 accounts CO₂ from the `"actual"` slice (`modules/52_carbon/normal_dec17/equations.gms:19`)"*). By contrast, Module 56's own pricing equation `q56_emis_pricing_co2` reads `pcm_carbon_stock(...,"actual")` for the previous stock but `vm_carbon_stock(...,"%c56_carbon_stock_pricing%")` for the current stock — a config-driven slice that **defaults to `actualNoAcEst`** (`modules/module_56.md:38,85,114-116`; `config/default.cfg:1835` per `cross_module/carbon_balance_conservation.md:101`). Net effect: in a default run, the CO2 figure Module 52 *reports* and the CO2 figure Module 56 *prices* are computed from two different `stockType` slices of the same underlying variable.

## `pcm_carbon_stock` vs. `vm_carbon_stock` across timesteps

Both are declared in Module 56 with identical dimensions `(j,land,c_pools,stockType)` (`modules/module_52.md:416,427`).

- **`vm_carbon_stock`** is the **current**-timestep carbon stock (mio. tC), populated *during* the optimization solve by the land modules' own carbon equations (see next section).
- **`pcm_carbon_stock`** is the **previous**-timestep carbon stock, produced by a **carry-forward** after each solve: the realized `vm_carbon_stock.l` values are copied into `pcm_carbon_stock` for use as the "previous" term in the *next* timestep's `q52_emis_co2_actual` (and `q56_emis_pricing_co2`).

🟢 The carry-forward is **split by carbon pool across two different modules' postsolve phases** (`modules/module_52.md:431`, `modules/module_56.md:581`):
- Above-ground pools (`ag_pools` = vegc, litc): carried forward in **Module 56** — `modules/56_ghg_policy/price_aug22/postsolve.gms:8`.
- Soil pool (`soilc`): carried forward in **Module 59 (SOM)** — `modules/59_som/cellpool_jan23/postsolve.gms:13` (also `static_jan19/postsolve.gms:9` for the alternate SOM realization).

⚠️ **Historical bug, now fixed**: the soil-pool carry-forward line was only added in develop commit `931db85c4` (2026-06-25). Before that, `pcm_carbon_stock(...,"soilc",...)` stayed frozen at its `preloop` initialization value instead of updating each timestep, so the soil term in `q52_emis_co2_actual` computed a *cumulative-since-init* stock change divided by a single timestep length rather than a true per-timestep flux (soil pools off by ~17x in a default H12 run per `modules/module_59_notes.md:15`). This did not affect default runs because the default GHG policy (`c56_emis_policy = reddnatveg_nosoil`) excludes soil from priced/reported emissions — but it would have affected any soil-inclusive policy variant or direct per-pool GDX reads.

## Which modules populate the cropland and SOM slices of `vm_carbon_stock`

🟢 **Cropland slice** (`land = "crop"`): **Module 29 (Cropland)**, realization `detail_apr24`, via equation `q29_carbon` (`modules/29_cropland/detail_apr24/equations.gms:38-42`):
```gams
q29_carbon(j2,ag_pools,stockType)..
  vm_carbon_stock(j2,"crop",ag_pools,stockType) =e=
    vm_carbon_stock_croparea(j2,ag_pools)
    + vm_fallow(j2) * sum(ct, fm_carbon_density(ct,j2,"crop",ag_pools))
    + m_carbon_stock_ac(v29_treecover, p29_carbon_density_ac, "ac", "ac_sub");
```
This is the sum of three terms: croparea carbon (folded in from **Module 30**'s separate interface variable `vm_carbon_stock_croparea(j2,ag_pools)`, populated by Module 30's own equation — Module 30 does *not* write `vm_carbon_stock` directly), fallow-land carbon (fallow area × `fm_carbon_density` from Module 52), and cropland tree-cover carbon (age-class carbon via the `m_carbon_stock_ac` macro, using `p29_carbon_density_ac`, which is itself seeded from Module 52's uncalibrated secdforest/plantation carbon-density curves per `s29_treecover_plantation`). Note this only covers the `ag_pools` (vegc, litc) — cropland `soilc` is not part of `q29_carbon`.

🟢 **SOM slice** (`c_pools = "soilc"`, **for all land types**, not just cropland): **Module 59 (SOM)**, realization `cellpool_jan23`, via equation `q59_carbon_soil` (`modules/59_som/cellpool_jan23/equations.gms:62`; also shown at `modules/module_59.md:176-177`):
```gams
q59_carbon_soil(j2,land,stockType) ..
  vm_carbon_stock(j2, land,"soilc",stockType) =e= ...
```
Module 59 is the **sole** writer of the `soilc` pool across *every* land type (crop, past, forestry, primforest, secdforest, urban, other) — confirmed by the cross-module summary table `modules/module_59.md:1068`: *"`vm_carbon_stock(j,land,"soilc",stockType)` — the `soilc` slice only | **M52** (carbon), **M56** (GHG policy) [readers] | populated at `cellpool_jan23/equations.gms:62`"*. (Module 52 itself supplies the *underlying* `fm_carbon_density` LPJmL data that Module 59 uses to derive things like subsoil carbon density, but Module 52 never writes to `vm_carbon_stock` — it only reads it.)

For completeness, the other slices: Module 31 (pasture), Module 32 (forestry), Module 34 (urban, fixed to 0), and Module 35 (natveg: primforest/secdforest/other) populate the `ag_pools` of their respective land types; Module 58 (peatland) does **not** populate `vm_carbon_stock` at all (`modules/module_52.md:424,584`).

## How Module 56 consumes Module 52's `vm_emissions_reg` — in one step

Module 56 consumes Module 52's output slice `vm_emissions_reg(i,emis_oneoff,"co2_c")` **for reporting only**, in a single postsolve assignment that copies the solved level into the module's own output parameter — it is **not** read by any equation:

```gams
ov_emissions_reg(...) = vm_emissions_reg.l(...);
```
🟢 `modules/56_ghg_policy/price_aug22/postsolve.gms:27` (cited at `modules/module_52.md:443`).

Module 56's *pricing* equation, `q56_emis_pricing_co2` (`modules/56_ghg_policy/price_aug22/equations.gms:19-22`), does **not** read `vm_emissions_reg` at all — it independently recomputes the CO2 flow directly from `pcm_carbon_stock`/`vm_carbon_stock`, running in parallel with (not downstream of) Module 52's calculation:
```gams
q56_emis_pricing_co2(i2,emis_oneoff) ..
  v56_emis_pricing(i2,emis_oneoff,"co2_c") =e=
                 sum((cell(i2,j2),emis_land(emis_oneoff,land,c_pools)),
                 (pcm_carbon_stock(j2,land,c_pools,"actual") - vm_carbon_stock(j2,land,c_pools,"%c56_carbon_stock_pricing%"))/m_timestep_length);
```
(`modules/module_56.md:82-85`). Module 56's other read of `vm_emissions_reg` (`equations.gms:17`, for the annual-gas pricing equation `q56_emis_pricing`) is restricted to the disjoint `emis_annual` slice, which structurally excludes `emis_oneoff`/`"co2_c"` (`core/sets.gms:314,320-323`; `modules/module_56.md:66`) — so that read never touches Module 52's output either.
