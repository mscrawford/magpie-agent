## (a) `vm_feed_balanceflow` for ruminants on pasture

Set by equation **`q70_feed_balanceflow`** (`modules/module_70.md:99-133`, formula at `equations.gms:25-29`), which applies only to the `kli_rum` × `"pasture"` slice (ruminant products = `livst_rum`, `livst_milk`; feed item = pasture only):

```
vm_feed_balanceflow(i2,kli_rum,"pasture") =e=
    (sum(ct, fm_feed_balanceflow(ct,i2,kli_rum,"pasture")))
        $ (p70_endo_scavenging_flag(i2,kli_rum) = 0)
    - (vm_prod_reg(i2,kli_rum) * sum(ct, im_feed_baskets(ct,i2,kli_rum,"pasture"))
        * s70_scavenging_ratio)
        $ (p70_endo_scavenging_flag(i2,kli_rum) > 0)
```

Two mutually exclusive branches, gated by `p70_endo_scavenging_flag(i,kli_rum)`:

1. **Flag = 0** — use the exogenous historical balance flow `fm_feed_balanceflow(t,i,kli_rum,"pasture")` (FAO-inventory-derived correction term).
2. **Flag > 0** — switch to an endogenous scavenging term: `-vm_prod_reg × im_feed_baskets × s70_scavenging_ratio`, i.e. the balance flow becomes a *negative* adjustment equal to 38.5% (`s70_scavenging_ratio` default, `input.gms:27`) of the baseline pasture requirement, reducing net pasture demand to represent scavenging/roadside grazing that substitutes for conventional pasture feed.

This is only possible for the ruminant-pasture slice because the presolve step relaxes the variable's bounds there specifically: `vm_feed_balanceflow.up(i,kli_rum,"pasture") = Inf` and `.lo(...) = -Inf` (`presolve.gms:10-11`, cited `modules/module_70.md:405-410`). Every other `(kap,kall)` combination — i.e. non-pasture feed items, and pasture for non-ruminants — is instead **fixed** to the historical FAO value: `vm_feed_balanceflow.fx(i,kap,kall) = fm_feed_balanceflow(t,i,kap,kall)` (`presolve.gms:9`, `modules/module_70.md:399-403`). So ruminant pasture is the *only* place `vm_feed_balanceflow` is a genuine decision variable rather than a fixed parameter passthrough.

`vm_feed_balanceflow` in turn enters feed demand via **`q70_feed`** (`equations.gms:17-20`, `modules/module_70.md:76-95`):
```
vm_dem_feed(i2,kap,kall) =g=
    vm_prod_reg(i2,kap) * sum(ct, im_feed_baskets(ct,i2,kap,kall))
    + sum(ct, vm_feed_balanceflow(i2,kap,kall))
```

## (b) Endogenous scavenging flag and its division-by-zero guard

**Flag calculation** (`presolve.gms:14-20`, formula given at `presolve.gms:17`, `modules/module_70.md:412-417`):
```
p70_endo_scavenging_flag(i,kli_rum) =
    -fm_feed_balanceflow(t-1,i,kli_rum,"pasture") / pc70_dem_feed_pasture(i,kli_rum)
```
If this ratio ≥ `s70_scavenging_ratio` (0.385 default), the flag is set to the ratio value; otherwise the flag is 0 (`modules/module_70.md:417`). Once set to a nonzero value, the flag persists — it is a parameter written once (not reset each timestep, since the outer timing guard, below, never fires again).

**What prevents division by zero**: the denominator `pc70_dem_feed_pasture(i,kli_rum)` is never allowed to reach exactly zero, via two coordinated mechanisms both documented under §Computational Notes (`modules/module_70.md:1181-1230`):

1. **Initialization**: `pc70_dem_feed_pasture(i,kli_rum) = 0.001` in `preloop.gms:10` — "small positive initial value prevents division by zero in scavenging flag calculation (`presolve.gms:17`)" (`modules/module_70.md:1183-1187`).
2. **Postsolve floor**: after every solve, `pc70_dem_feed_pasture(i,kli_rum) = max(vm_dem_feed.l(i,kli_rum,"pasture"), 0.001)` (`postsolve.gms:9`, `modules/module_70.md:1225-1229`) — "Maximum ensures nonzero denominator" going into the *next* timestep's flag calculation.

So the guard is a floor applied at two points in the parameter's lifecycle (initial value, and every postsolve update), not a single runtime check inside the division itself.

**Timing** (`presolve.gms:14-20`, detailed at `modules/module_70.md:419-445`): the flag calculation is gated by a nested `if` — an outer guard restricting it to timesteps that are themselves historical (`t ∈ t_past`), and an inner guard narrowing that to the *last* historical timestep with future timesteps still ahead. Under the default config (`c_past = "till_2015"` → `t_past = {y1965…y2015}`; `c_timesteps = "coup2100"`), this resolves to **`t = y2015`** — the flag is computed once, at the historical/future boundary, and then frozen for all subsequent (future) timesteps. `module_70.md` flags (in a self-correction dated 2026-07-17) that this is the *opposite* of "first future timestep" phrasing that appeared in an earlier version of the doc — the switch to endogenous scavenging takes effect **within** the last historical timestep itself, not after it.

## (c) `c70_feed_scen` default and what it selects

**Switch**: `c70_feed_scen = "ssp2"` (`input.gms:8`, `modules/module_70.md:897-903`; also listed under Configuration Options, `modules/module_70.md:365-370, 895-903`).

**What it selects**: the feed-basket trajectory scenario dimension `feed_scen70` on the input table `f70_feed_baskets(t_all,i,kap,kall,feed_scen70)` (`input.gms:36-39`). Legal values (`sets.gms:16-18`): `ssp1, ssp2, ssp3, ssp4, ssp5`, SDP variants `SDP, SDP_EI, SDP_MC, SDP_RC`, and `constant`. Each option reflects a different SSP/SDP intensification storyline that determines feed conversion efficiency and feed composition (concentrate vs. roughage share) as livestock productivity changes over the projection period.

**Selection logic** (`preloop.gms:13-23`, `modules/module_70.md:368-372`):
- Historical period (`t ≤ sm_fix_SSP2`): always uses the SSP2 feed baskets, regardless of the switch setting.
- Future period: uses whichever scenario `c70_feed_scen` selects (default `ssp2`, so under default config there is no discontinuity at the historical/future boundary).

## Connection to Modules 16 and 17

**M70 → M16**: `vm_dem_feed(i,kap,kall)` (declared `declarations.gms:11`, mio. tDM/yr) is the module's headline output. Module 16 has no single monolithic "total demand" equation; instead it has one supply-balance equation per product-type group, and `vm_dem_feed` is summed into each of the non-forestry ones via `sum(kap4, vm_dem_feed(...))`:
- `q16_supply_crops` (`modules/module_16.md:66-96`; formula `equations.gms:19-29`):
  ```
  vm_supply(i2,kcr) =e=
      vm_dem_food(i2,kcr) + sum(kap4, vm_dem_feed(i2,kap4,kcr))
      + vm_dem_processing(i2,kcr) + vm_dem_material(i2,kcr)
      + vm_dem_bioen(i2,kcr) + vm_dem_seed(i2,kcr)
      + v16_dem_waste(i2,kcr) + sum(ct, f16_domestic_balanceflow(ct,i2,kcr))
  ```
- The same `sum(kap4, vm_dem_feed(...))` term appears in `q16_supply_livestock`, `q16_supply_secondary`, `q16_supply_residues`, and `q16_supply_pasture` (`equations.gms:22,34,43,53,63` per `modules/module_16.md:349`).

`vm_supply` (declared `declarations.gms:11`) is the aggregated demand-side output that Module 21 (Trade) balances against production — described in `module_16.md` as the module's "ONLY equation-level consumer" interface into `q21_trade_glo` (`modules/module_16.md:565`, citing `modules/21_trade/selfsuff_reduced/equations.gms:14`).

**M17 → M70**: Module 17's core equation `q17_prod_reg` (`modules/module_17.md:43-53`; formula `equations.gms:10-11`) defines the regional production variable that M70 consumes:
```
q17_prod_reg(i2,k) ..
vm_prod_reg(i2,k) =e= sum(cell(i2,j2), vm_prod(j2,k));
```
`vm_prod_reg(i,kap)` is read directly by both `q70_feed_intake_pre` (`vm_prod_reg(i2,kap) * sum(ct, im_feed_baskets(...))`) and `q70_feed` (same term, plus the balance-flow addition described in (a)), i.e. it drives feed demand from the production side (`modules/module_70.md:35-37, 82-84`).

**The loop closes through trade**: `vm_prod_reg` also feeds Module 21's global trade balance (`q21_trade_glo: sum(i,vm_prod_reg) =g= sum(i,vm_supply) + balanceflow`, cited at `modules/module_17.md:281, 770-772`), where `vm_supply` is the M16 aggregate that already contains `vm_dem_feed`. `module_70.md` names this explicitly as a potential circular dependency (`modules/module_70.md:1278-1282`):

> "Module 70 (feed demand) → Module 16 (demand aggregation) → Module 21 (trade) → Module 17 (production) → Module 70 (feed demand)."
>
> **Resolution** (as stated in the doc): `vm_prod_reg(i,kap)` is treated as the production side of a joint optimization — the `q70_feed` inequality (`=g=`, not `=e=`) lets feed demand exceed the strict production×basket minimum rather than forcing an equality, so the solver can find a simultaneous equilibrium across `q70_feed`, `q16_supply_*`, `q21_trade_glo`, and `q17_prod_reg` within a single timestep's solve, rather than needing this resolved via any exogenous ordering.

Note this differs from the *lagged*, presolve-computed M70↔M14 pasture-management cycle described elsewhere in `module_70.md` (`pm_past_mngmnt_factor`, a parameter fixed before the solve) — the M70↔M16↔M21↔M17 loop above is resolved **simultaneously within the LP/NLP solve**, since every variable in it (`vm_dem_feed`, `vm_supply`, `vm_prod_reg`, `vm_prod`) is a GAMS decision variable, not a presolve-fixed parameter.
