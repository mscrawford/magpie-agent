## 1. Cost formation in Module 70

### 1.1 The factor-requirement input: `i70_fac_req_livst`

Both the labor and capital cost equations (and the fish equation, in simplified form) build on a single per-unit factor-requirement parameter, computed in `preloop.gms:88` (per `modules/module_70.md:155-160`):

```
i70_fac_req_livst(t,i,kli) = i70_cost_regr(i,kli,"cost_regr_b")
    * sum(sys_to_kli(sys,kli), i70_livestock_productivity(t,i,sys))
    + i70_cost_regr(i,kli,"cost_regr_a")
```

This is a linear regression: per-unit factor cost = slope (`cost_regr_b`) × livestock productivity (`i70_livestock_productivity`, ton FM/animal/yr) + intercept (`cost_regr_a`). Units: USD17MER per tDM.

### 1.2 The cost-regression coefficients: `i70_cost_regr` and where they come from

`i70_cost_regr(i,kap,"cost_regr_a"/"cost_regr_b")` is populated in `preloop.gms:82,83,86` from the input table `f70_cost_regr(kap,cost_regr)` (`input.gms:51-54`), itself sourced from the file `f70_capit_liv_regr.csv` (`modules/module_70.md:1140-1142`).

- **Data source**: GTAP 7 database (Narayanan & Walmsley 2008), per `equations.gms:48-49` / `modules/module_70.md:562`.
- **Slope** (`cost_regr_b`): `i70_cost_regr(i,kap,"cost_regr_b") = f70_cost_regr(kap,"cost_regr_b")` (`preloop.gms:86`). Ruminants (beef, milk) get `b > 0` — costs rise with intensification; non-ruminants (pork, poultry, eggs) and fish get `b = 0` — constant per-unit costs.
- **Intercept** (`cost_regr_a`), config switch `c70_fac_req_regr` (`input.gms:21-22`, default `"glo"`):
  - **`"glo"` (default)**: global GTAP intercept used directly — `i70_cost_regr(i,kli,"cost_regr_a") = f70_cost_regr(kli,"cost_regr_a")` (`preloop.gms:82`).
  - **`"reg"` (non-default)**: intercept recalibrated per-region so the regression matches the last historical year's actual costs (`preloop.gms:83`), using `f70_hist_factor_costs_livst` / `f70_hist_prod_livst` (`input.gms:72-82`). This path additionally overrides `i70_fac_req_livst` with actual historical values for years strictly after 1990 up to the last historical year (`preloop.gms:90`) — a range documentation note (R58) flags as strictly `> 1990`, not `≥ 1990`.

### 1.3 `vm_cost_prod_livst("labor")` — `q70_cost_prod_liv_labor`

`equations.gms:59-62` (per `modules/module_70.md:136-167`):

```
vm_cost_prod_livst(i2,"labor") =e=
    sum(kli, vm_prod_reg(i2,kli) * sum(ct, i70_fac_req_livst(ct,i2,kli)))
    * sum(ct, pm_factor_cost_shares(ct,i2,"labor"))
    * sum(ct, (1 / pm_productivity_gain_from_wages(ct,i2))
        * (pm_hourly_costs(ct,i2,"scenario") / pm_hourly_costs(ct,i2,"baseline")))
```

Production (`vm_prod_reg`, from Module 17) × factor requirement (§1.1) × labor's share of total factor cost (`pm_factor_cost_shares(i,"labor")`, from Module 38) × a wage-scenario adjustment (`pm_hourly_costs` scenario/baseline ratio and `pm_productivity_gain_from_wages`, both from Module 36 — Employment).

### 1.4 `vm_cost_prod_livst("capital")` — `q70_cost_prod_liv_capital`

**Default (`fbask_jan16`)**, `equations.gms:64-66`:

```
vm_cost_prod_livst(i2,"capital") =e=
    sum(kli, vm_prod_reg(i2,kli) * sum(ct, i70_fac_req_livst(ct,i2,kli)))
    * sum(ct, pm_factor_cost_shares(ct,i2,"capital"))
```

Same production × factor-requirement base as labor, times capital's factor-cost share — no wage adjustment, no capital-stock mechanism. **This default path does not use a capital-need parameter at all.**

**Alternative (`fbask_jan16_sticky`, non-default)** replaces this equation with an investment/annuity formulation, and this is where the capital-need parameter appears:

```
vm_cost_prod_livst(i2,"capital") =e=
    sum(kli, v70_investment(i2,kli))
    * ((1 - s70_depreciation_rate) * sum(ct, pm_interest(ct,i2) / (1 + pm_interest(ct,i2)))
       + s70_depreciation_rate)
```

with a companion equation `q70_investment`:

```
v70_investment(i2,kli) =g=
    vm_prod_reg(i2,kli) * sum(ct, p70_capital_need(ct,i2,kli))
    - sum(ct, p70_capital(ct,i2,kli))
```

**The capital-need parameter — `p70_capital_need`** (`fbask_jan16_sticky/presolve.gms`; exact line not given in the doc's citation — see Session-close notes):

```
p70_capital_need(t,i,kli) = i70_fac_req_livst(t,i,kli)
    * pm_factor_cost_shares(t,i,"capital")
    / (pm_interest(t,i) + s70_depreciation_rate)
    * s70_multiplicator_capital_need
```

It converts the same per-unit factor requirement `i70_fac_req_livst` (§1.1) — scaled by the capital cost share and multiplied by `s70_multiplicator_capital_need` (default 1) — into a capital **stock** requirement by dividing through `(pm_interest + s70_depreciation_rate)` (interest from Module 12; depreciation rate `s70_depreciation_rate`, default 0.05). `v70_investment` then covers the gap between this required stock and the pre-existing stock `p70_capital(t,i,kli)`, which is carried across timesteps via a two-phase presolve/postsolve lifecycle (`presolve.gms:83-95` initializes/depreciates; `postsolve.gms:13` accumulates realized investment into the next timestep — per `modules/module_70.md:272-292`).

**So: the capital-need parameter (`p70_capital_need`) directly reuses the factor-requirement input `i70_fac_req_livst` described in §1.1** — it is that same regression-based per-unit cost, repurposed via an annuity-style division by `(interest + depreciation)` to size a capital stock rather than a flow cost. This entire mechanism is confined to the non-default `fbask_jan16_sticky` realization.

### 1.5 `vm_cost_prod_fish(i)` — `q70_cost_prod_fish`

`equations.gms:68-70` (per `modules/module_70.md:196-213`; shared by both realizations):

```
vm_cost_prod_fish(i2) =e=
    vm_prod_reg(i2,"fish") * i70_cost_regr(i2,"fish","cost_regr_a")
```

Simple linear cost: fish production × the intercept-only regression coefficient (`i70_cost_regr(i,"fish","cost_regr_a")`, from GTAP via `preloop.gms:85`). No slope term (fish `b = 0`), no productivity scaling, no labor/capital split — a single, unfactored cost.

## 2. Every `vm_cost_*` variable Module 70 populates

Per `modules/module_70.md:817-822` (declarations at `declarations.gms:12-13`), Module 70 declares and populates exactly **two** `vm_cost_*` identifiers:

1. **`vm_cost_prod_livst(i,factors)`** — factors ∈ {labor, capital}; mio. USD17MER/yr. Populated by `q70_cost_prod_liv_labor` and `q70_cost_prod_liv_capital`.
2. **`vm_cost_prod_fish(i)`** — mio. USD17MER/yr. Populated by `q70_cost_prod_fish`.

**Naming trap to flag explicitly**: `vm_costs_additional_mon(i)` is **not** a Module 70 variable despite the superficial `vm_cost*` resemblance. Per `modules/module_70_notes.md:9-15` and `modules/module_11.md:505-509`, it is declared in **Module 71** (Livestock Disaggregation), specifically the default realization `foragebased_jul23` (`modules/71_disagg_lvst/foragebased_jul23/declarations.gms:11`), as a penalty cost for additionally-transported monogastric `livst_egg` in M71's regional-to-cell disaggregation — unrelated to M70's aggregate cost accounting. Both `modules/module_70.md:822` and `module_11.md:541` flag this as a distinct naming pattern (`vm_costs_` with an `s`, not `vm_cost_prod_`) precisely because it has been a documented confusion source.

## 4. How these costs enter Module 11's aggregation

Module 11 (`default` realization, the only realization — `modules/module_11.md:622-626`) is a **pure aggregator**, with no cost logic of its own (`modules/module_11.md:20`). Two equations:

**`q11_cost_reg(i2)`** (`equations.gms:15-47`) sums 32 terms (31 costs + 1 negative reward) into the regional intermediate `v11_cost_reg(i2)`. The two Module 70 outputs enter directly, unsummed over any extra dimension for fish and summed over `factors` for livestock:

```
... + vm_cost_prod_fish(i2)                          -- equations.gms:18
    + sum(factors,vm_cost_prod_livst(i2,factors))     -- equations.gms:19
... 
```

(per `modules/module_11.md:86-119`, full equation block, and the per-term citations at `modules/module_11.md:212` and `:222`).

**`q11_cost_glo`** (`equations.gms:10`) then sums `v11_cost_reg(i2)` across all regions into `vm_cost_glo`:

```
vm_cost_glo =e= sum(i2, v11_cost_reg(i2));
```

`vm_cost_glo` is the objective function variable. Per `modules/module_11.md:1192-1194`, Module 11 does not consume its own output — the actual consumer is **Module 80 (Optimization)**: `solve magpie USING nlp MINIMIZING vm_cost_glo;` (`modules/80_optimization/nlp_apr17/solve.gms:34`, default realization `nlp_apr17`). So the full chain for livestock/fish costs is:

```
i70_fac_req_livst (× production, × cost shares, × wage/interest adjustments)
  → vm_cost_prod_livst(i,"labor"/"capital"), vm_cost_prod_fish(i)   [Module 70]
  → q11_cost_reg → v11_cost_reg(i)                                  [Module 11]
  → q11_cost_glo → vm_cost_glo                                      [Module 11]
  → MINIMIZING vm_cost_glo (solve statement)                        [Module 80]
```

### Sign/declaration note

`vm_cost_prod_livst` and `vm_cost_prod_fish` are not flagged among Module 11's documented list of 9 "FREE variable" (sign-unconstrained) terms (`modules/module_11.md:549-565`) — that list is: `vm_processing_substitution_cost`, `vm_cost_landcon`, `vm_cost_transp`, `vm_cost_AEI`, `vm_p_fert_costs`, `vm_emission_costs`, `vm_reward_cdr_aff`, `vm_peatland_cost`, `vm_bioenergy_utility`. By elimination per that doc's own framing, the livestock/fish cost terms fall in the `positive variables` (non-negative) declaration class, consistent with them being straightforward production-cost accumulations with no reward/substitution/penalty semantics — though the doc does not state this affirmatively for these two variables, so treat it as an inference from the stated list rather than a directly documented fact.

### A related, undocumented-elsewhere feedback loop worth noting

The labor-cost equation (§1.3) reads Module 36 (Employment) wage parameters, and Module 36 in turn reads `vm_cost_prod_livst(i,"labor")` for its employment calculation (`modules/36_employment/exo_may22/equations.gms:24`, per `modules/module_70.md:188-192` and `:1548-1553`). This is a real M70↔M36 cycle the doc flags as a correction (R58) — relevant context if modifying either module's labor-cost logic, though it does not change how the cost enters Module 11's objective.
