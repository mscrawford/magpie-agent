## 1. Module 11's objective: how many cost terms, via what set and equation

Module 11 (`default` realization) builds the objective in two equations:

- **`q11_cost_glo`** (`modules/11_costs/default/equations.gms:10`):
  `vm_cost_glo =e= sum(i2, v11_cost_reg(i2));`
  This sums the regional cost `v11_cost_reg(i2)` over the regional set **`i2`** (active MAgPIE regions, typically 10-14) to produce `vm_cost_glo` — the single variable MAgPIE minimizes.

- **`q11_cost_reg`** (`modules/11_costs/default/equations.gms:15-47`):
  `v11_cost_reg(i2) =e= sum(factors,vm_cost_prod_crop(i2,factors)) + ... - vm_reward_cdr_aff(i2) + ...`
  This is a flat sum of **32 cost-term components** (31 added with `+`, 1 subtracted: `vm_reward_cdr_aff`, the afforestation CDR revenue term) drawn from 27 other modules. There is no single GAMS *set* whose members enumerate the 32 terms — each term is a distinct interface variable named explicitly in the equation body; only the *internal* dimensions of individual terms (e.g. `factors`, `kres`, `cell(i2,j2)×land`, `cell(i2,j2)×k`) are summed via real sets. `v11_cost_reg` itself is declared at `declarations.gms:10`.

So: the objective (`vm_cost_glo`) aggregates **regions** via the set `i2` in `q11_cost_glo`; the regional cost (`v11_cost_reg`) aggregates **32 cost terms** via the hardcoded list in `q11_cost_reg`. Module 11 has no input files or switches of its own — it is a pure aggregator (`module_11.md` §1.2, §6).

Two of those 32 terms are supplied by Module 56 and are directly relevant below: `vm_emission_costs(i)` (positive, `equations.gms:26`) and `vm_reward_cdr_aff(i)` (negative, `equations.gms:27`).

## 2. Module 59: how soil carbon populates `vm_carbon_stock`

`cellpool_jan23`'s equation **`q59_carbon_soil`** (`modules/59_som/cellpool_jan23/equations.gms:61-64`):

```gams
q59_carbon_soil(j2,land,stockType) ..
  vm_carbon_stock(j2, land,"soilc",stockType) =e=
    v59_som_pool(j2, land) +
    vm_land(j2, land) * sum(ct,i59_subsoilc_density(ct,j2));
```

Module 59 writes only the **`"soilc"`** slice of `vm_carbon_stock` (a 4-D variable `vm_carbon_stock(j,land,c_pools,stockType)` that is **declared in Module 56**, `modules/56_ghg_policy/price_aug22/declarations.gms:34` — M59 is a populator, not the declarer; `module_56_notes.md` flags this producer/declarer split explicitly as a recurring confusion). The equation is indexed over the free `stockType` set (`"actual"`/`"actualNoAcEst"`), so both slices get filled identically by this equation — M59 does not itself distinguish stock types.

The value is **topsoil pool (dynamic) + subsoil reference (static)**:
- `v59_som_pool(j,land)` — the actual, dynamically-converging topsoil pool from `q59_som_pool` (`equations.gms:46-52`), itself a weighted blend of the equilibrium target (`v59_som_target`, from `q59_som_target_cropland`/`q59_som_target_noncropland`) and legacy carbon from land-use transitions, moving 15%/yr toward equilibrium (`i59_lossrate(t) = 1 - 0.85^m_yeardiff(t)`, `preloop.gms:45`).
- `i59_subsoilc_density(t,j)` — a static-by-land-use-effect parameter derived from `fm_carbon_density(t,j,"other","soilc") - f59_topsoilc_density(t,j)` (`preloop.gms:12`); not frozen in time (varies with climate under the default `c59_som_scenario="cc"`), but not responsive to land-use/management.

Other land pools (`vegc`, `litc`) and the crop/pasture/forestry/natveg slices of `soilc`'s companions are populated by Modules 29 (crop, folding in M30's `vm_carbon_stock_croparea`), 31 (pasture), 32 (forestry), 34 (urban, fixed to 0), and 35 (primforest/secdforest/other) — M59 owns *only* the `soilc` slice, for *all* land types (`module_59.md` §"Provides to"/Interface Variables table).

## 3. `pcm_carbon_stock` at the start of a timestep, and how it differs from `vm_carbon_stock`

**`pcm_carbon_stock`** is a GAMS **parameter** (fixed data, `p`-prefix), not a decision variable. It represents the **realized carbon stock carried over from the end of the previous timestep's solve** — i.e., the reference "previous stock" available at the start of the current timestep, before this period's land-use optimization runs.

- **Population mechanism (soil slice)**: Module 59's `postsolve.gms:13` (`cellpool_jan23`; also `static_jan19/postsolve.gms:9`):
  `pcm_carbon_stock(j,land,"soilc",stockType) = vm_carbon_stock.l(j,land,"soilc",stockType);`
  i.e., after each solve, the *solved level* of the current timestep's `vm_carbon_stock` (soil slice) is copied into `pcm_carbon_stock`, becoming next timestep's "previous" reference.
- The above-ground pools (`ag_pools` = vegc, litc) are carried forward the same way but by **Module 56's own** `postsolve.gms:8` — the carry-forward is split by pool, mirroring the split in who populates `vm_carbon_stock` (`module_59.md` §3.4).
- **Initialization** (first timestep only): M59's `preloop.gms:30-35` seeds the soil slice from `pc59_som_pool` + `i59_subsoilc_density` at `"y1995"`; M56's `preloop.gms:10-11` seeds the `ag_pools` slice.
- `pcm_carbon_stock` is declared in Module 56 (`module_59.md` explicitly: "`pcm_carbon_stock` is declared in Module 56 and consumed as the 'previous stock' term by `q52_emis_co2_actual` (Module 52) and `q56_emis_pricing_co2` (Module 56)").

**Difference from `vm_carbon_stock`**:
| | `pcm_carbon_stock` | `vm_carbon_stock` |
|---|---|---|
| GAMS kind | Parameter (fixed, exogenous to the current solve) | Free/positive variable (endogenous, solved each timestep) |
| Timing | Previous timestep's *solved* stock — the state at the **start** of the current timestep | Current timestep's stock — determined by **this** timestep's optimized land allocation |
| Who sets it | Copied post-solve from `vm_carbon_stock.l` of the prior timestep | Computed live by the populating equations (M29/31/32/34/35/59) during the current solve |
| stockType used downstream | Always read as `"actual"` in `q56_emis_pricing_co2` (hardcoded) | Read as the *switch-controlled* slice `%c56_carbon_stock_pricing%` (default `actualNoAcEst`) |

That last row is a real asymmetry: the "previous" reference is always the `"actual"` stock, but the "current" comparator can be a different `stockType` slice depending on `c56_carbon_stock_pricing` — so the two operands of the pricing equation are not guaranteed to be the same stock definition (`cross_module/carbon_balance_conservation.md` §2.3 flags this explicitly: "the priced CO₂ is therefore not computed from the same stock slice as the reported CO₂").

**Bug-fix history worth flagging**: until MAgPIE develop commit `931db85c4` (2026-06-25), the soil-slice carry-forward line in `postsolve.gms` was **absent**, so `pcm_carbon_stock(...,"soilc",...)` stayed frozen at its `preloop` initialization instead of updating each timestep — meaning the soil term in the CO2-pricing/accounting equations computed a cumulative-since-1995 change divided by one timestep length (soil pools off by ~17x in a default H12-length run), not a true per-timestep flux. Default runs were unaffected because the default GHG policy (`reddnatveg_nosoil`) excludes soil carbon from pricing, and default soil reporting routes through `magpie4::emisSOC` rather than this pathway (`module_59_notes.md` item 4; `module_59.md` §3.4 warning box).

## 4. The chain to Module 56's GHG pricing

`price_aug22`'s **`q56_emis_pricing_co2`** (`modules/56_ghg_policy/price_aug22/equations.gms:19-22`):

```gams
q56_emis_pricing_co2(i2,emis_oneoff) ..
  v56_emis_pricing(i2,emis_oneoff,"co2_c") =e=
    sum((cell(i2,j2),emis_land(emis_oneoff,land,c_pools)),
    (pcm_carbon_stock(j2,land,c_pools,"actual") - vm_carbon_stock(j2,land,c_pools,"%c56_carbon_stock_pricing%"))/m_timestep_length);
```

This reads `pcm_carbon_stock` and `vm_carbon_stock` **directly**, bypassing `vm_emissions_reg` entirely (`module_56.md` §2.2 calls this out as a deliberate architectural choice, in contrast to the CH4/N2O path which does route through `vm_emissions_reg` via `q56_emis_pricing`, `equations.gms:15-17`). Because `emis_land(emis_oneoff,land,c_pools)` spans all land×pool combinations including `soilc`, M59's soil contribution is mechanically included in this sum.

Full chain from M59's soil-carbon write-out to the objective:

1. **M59** `q59_carbon_soil` (`cellpool_jan23/equations.gms:61-64`) populates `vm_carbon_stock(j,land,"soilc",stockType)` for the current timestep.
2. **M59** `postsolve.gms:13` copies the solved `vm_carbon_stock.l(...,"soilc",...)` into `pcm_carbon_stock(...,"soilc",...)`, becoming next timestep's "previous stock."
3. **M56** `q56_emis_pricing_co2` (`price_aug22/equations.gms:19-22`) computes `(pcm_carbon_stock − vm_carbon_stock)/m_timestep_length` → `v56_emis_pricing(i2,emis_oneoff,"co2_c")`, for every land×pool combination in `emis_oneoff`, including the soil ones.
4. **M56** `q56_emission_cost_oneoff` (`equations.gms:45-52`) multiplies `v56_emis_pricing` by `m_timestep_length`, the pollutant price `im_pollutant_prices`, and the annuity factor `pm_interest/(1+pm_interest)` → `v56_emission_cost(i2,emis_oneoff)`.
5. **M56** `q56_emission_costs` (`equations.gms:56-58`) sums `v56_emission_cost` over `emis_source` → `vm_emission_costs(i2)` (`declarations.gms:39`).
6. **M11** `q11_cost_reg` (`equations.gms:26`) adds `vm_emission_costs(i2)` as one of its 32 terms → `v11_cost_reg(i2)` → **M11** `q11_cost_glo` (`equations.gms:10`) sums over regions → `vm_cost_glo`, the objective.

**Default-run caveat (important)**: step 4's multiplier `im_pollutant_prices` is zeroed out for soil-carbon emission sources under the **default** `c56_emis_policy = "reddnatveg_nosoil"` (its policy matrix `f56_emis_policy` has `co2_c` entries = 0 for all `*_soilc` sources — "nosoil" is literally what the name encodes; `module_56.md` §3.7, §5.2). So in a default run, step 3's soil `v56_emis_pricing` value is still computed (the stock-difference math runs regardless), but it is priced at **zero**, contributing nothing to `vm_emission_costs` and hence nothing to the objective through this path. Soil-inclusive policies (e.g. `"all"`, `"all_nosoil"` — despite the name, `all_nosoil` *does* include CO2 LULUCF but not soil either; `"sdp_all"` does include soil) would activate this pathway; so would flipping `c56_carbon_stock_pricing`/`c56_emis_policy` off default.

This also explains why the pre-fix carry-forward bug (§3 above) was benign for default runs: with the soil price fixed at zero, an incorrect `pcm_carbon_stock` soil value fed into `q56_emis_pricing_co2` still multiplied out to zero cost — the bug was live in the accounting math but silent in the objective, under defaults.
