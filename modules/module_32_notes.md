## вљ пёЏ Warnings & Common Mistakes

### `npi_ndc_aff_pol.cs3` is REGENERATED at run start вЂ” it is not a fixed download

This is the single most misleading thing about `c32_aff_policy`, and it is easy to get exactly backwards. Read this before reasoning about any afforestation-policy input.

**The GAMS side has no code branch at all.** The entire policy selection is one table lookup:

```gams
p32_aff_pol(t,j) = round(f32_aff_pol(t,j,"%c32_aff_policy%"),6);
```
(`modules/32_forestry/dynamic_may24/preloop.gms:182`)

`c32_aff_policy` appears in GAMS *only* at `input.gms:9` (`$setglobal`) and that line. There are **zero** `$if` / `$ifthen` branches on the policy name anywhere. So a policy's whole substance is a **column** of `npi_ndc_aff_pol.cs3` (`input.gms:73`).

**But that `.cs3` is a generated artifact, not an input you download.** It is gitignored (`.gitignore:7`, `*.cs*`) and **rewritten at run start by the R layer**:

| Step | Where |
|---|---|
| Default config enables the check | `config/default.cfg:123` вЂ” `cfg$recalc_npi_ndc <- "ifneeded"` (**this is the default**) |
| Missing-column guard | `scripts/start_functions.R:380-383` вЂ” `affexp_missing`, `ndcdelay_missing` test whether the selected policy has a column in the current file |
| Guard forces regeneration | `scripts/start_functions.R:385-391` вЂ” sets `cfg$recalc_npi_ndc <- TRUE` |
| Regenerator overwrites the file GAMS includes | `scripts/npi_ndc/start_npi_ndc.R:22` вЂ” `outfolder_aff` includes `../../modules/32_forestry/input/` |

Where each policy's substance actually lives:
- **`affexp`** вЂ” rows in `scripts/npi_ndc/policies/policy_definitions.csv` (delivered by the input download).
- **`ndcdelay`** вЂ” needs no input rows at all; it is **derived in R** from the existing `ndc` rows by shifting target years в‰Ґ2030 by a country-specific PRISMA delay (`start_npi_ndc.R`).

**So a stale header proves nothing.** If you `head` the on-disk `.cs3` and see only `dummy,dummy,none,npi,ndc`, that is a *stale artifact from a previous run* вЂ” not evidence that `affexp` / `ndcdelay` are unusable. Under the default config the file is regenerated with the needed column before GAMS ever reads it. Both policies are wired into named scenario presets (`config/scenario_config.csv:51`).

**The one real footgun (narrow, non-default):** the whole guard lives inside `if(cfg$recalc_npi_ndc=="ifneeded")` (`scripts/start_functions.R:375`). If you override **`cfg$recalc_npi_ndc <- FALSE`**, no regeneration happens, the stale `.cs3` is `$include`d as-is, and GAMS **zero-fills an absent table column with no error and no warning** вЂ” so `f32_aff_pol(t,j,"ndcdelay") = 0`, `p32_aff_pol = 0` everywhere, and your "policy" run is silently identical to `none`. GAMS only errors on the converse (an *unknown* column label в†’ domain violation). If you have disabled recalculation, check the header yourself:

```bash
head -5 modules/32_forestry/input/npi_ndc_aff_pol.cs3 | grep -v '^\*'
```

### рџљ© How this note got written вЂ” a cautionary tale worth keeping

The first version of this file asserted the opposite: that `ndcdelay` was **broken** вЂ” "selectable but unfed", silently producing a zero policy вЂ” and recommended reporting it to the PRISMA authors. Every *premise* was true and verified (no GAMS branch; the on-disk `.cs3` really does have only 3 policy columns; `affexp`/`ndcdelay` really do appear 0 times in its 6405 lines; neither feature commit bumped `cfg$input`). The **conclusion was false**, because the reasoning never left the GAMS layer: it inferred the provenance of a *gitignored, regenerated* artifact from git history alone.

The guards that refute it (`affexp_missing`, `ndcdelay_missing`) were added **by the very commits being accused** вЂ” `a54cd02c6` and `58bde5788` each modified `scripts/start_functions.R`. The author had already handled the exact failure mode.

**The transferable lesson:** a file under `modules/*/input/` is **not** necessarily an input you were given. It may be produced by `scripts/` at run start. **Before claiming an input is missing or stale, grep `scripts/` for the switch name and for the file name вЂ” not just `modules/` and `git log`.** One `grep -rn "c32_aff_policy" .` would have surfaced `scripts/start_functions.R:380` immediately.

## рџ’Ў Lessons Learned

- 2026-07-14: `c32_aff_policy` is a pure data lookup with no GAMS code branch (`preloop.gms:182`) вЂ” but the file it looks into (`npi_ndc_aff_pol.cs3`) is **regenerated by the R start layer** whenever the selected policy's column is missing (`scripts/start_functions.R:380-391`, default `cfg$recalc_npi_ndc <- "ifneeded"`). Do not reason about MAgPIE input files from `modules/*/input/` contents plus git history alone: `*.cs*` is gitignored and several inputs are run-time products. (Source: session experience вЂ” an agent-generated "silent zero" bug report that was refuted by an adversarial audit.)
- 2026-07-14: GAMS zero-fills a `table` column that is declared in the set but absent from the `$ondelim` `.cs3` header вЂ” no error, no warning. That IS a genuine silent-failure mode; it is simply not reachable here under the default config, only if you set `cfg$recalc_npi_ndc <- FALSE`. (Source: session experience.)
